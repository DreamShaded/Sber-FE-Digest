## 🎨 CSS

- [UI без брейкпоинтов: intrinsic layout на практике](https://frontendmasters.com/blog/building-a-ui-without-breakpoints/). Брейкпоинты решали задачу создания адаптивных интерфейсов, но компонентный веб их перерос: один и тот же компонент живёт в сайдбаре, модалке, дашборде и фул-вид ленте одновременно, и viewport-ширина не всегда лучшая переменная для выбора лэйаута. Автор показывает, как заменить цепочки `@media` intrinsic-примитивами: `auto-fit` + `minmax()` вместо четырёх брейкпоинтов для грида, `clamp()` для плавной типографики, container queries для компонентов, которые должны реагировать на своего родителя, а не на экран. Media queries остаются для более узких настроек, например, выключения анимации.
- - [Новые свойства CSS multi-column: column-wrap и column-height](https://css-tricks.com/css-multi-column-layout-wrapping-features/). Chrome 145 добавляет новые CSS свойства: `column-wrap: wrap` заставляет лишние колонки переноситься в новую строку вниз, `column-height` задаёт явную высоту ряда. Связка с `scroll-snap-type: y mandatory` даёт вертикальный пагинатор в духе журнала без JS. Есть и ограничения: динамический контент без фиксированной высоты делает результат непредсказуемым, и поддержка пока только в Chrome 145+.

## ⚡ JS / TS

- [Pretext: библиотека, которая считает текстовую вёрстку в 500 раз быстрее браузера. Зачем это нужно и как работает](https://habr.com/ru/articles/1020058/). Каждое обращение к DOM через getBoundingClientRect или offsetHeight запускает reflow - и всё начинает дёргаться. Pretext делает иначе: один раз замеряет ширину слов через Canvas API (`prepare()`), дальше считает переносы и высоту чистой арифметикой (`layout()`) — без DOM, без блокировок, ~0.1 мс на 500 текстов. Поддерживает CJK, арабский, эмодзи, bidirectional-текст. 15 КБ, ноль зависимостей, работает везде. Не заменяет DOM-рендеринг, заменяет измерение текста.
- [Модульная система JavaScript как архитектурное решение](https://css-tricks.com/the-javascript-module-system-architecture/). Статья объясняет, почему ESM сознательно пожертвовал гибкостью CommonJS: `require()` можно вызвать где угодно, в `if` или цикле, поэтому статические инструменты не могут понять граф зависимостей до рантайма и tree-shaking не работает. ESM с декларациями строго вверху файла решает это. Дальше автор переходит к Clean Architecture и правилу зависимостей Роберта Мартина: зависимости должны течь только внутрь, к бизнес-логике, а не наружу к фреймворкам.
- [Обзор актуального состояния языка ECMAScript 2025 от Криса Койера](https://frontendmasters.com/blog/what-to-know-in-javascript-2026-edition/). ES16 принёс Iterator Helpers с ленивыми вычислениями (никаких промежуточных массивов), методы для работы с двумя Set-ами сразу (`intersection`, `difference`, `union`), многострадальный `RegExp.escape()` после 15 лет ожиданий и `Promise.try()` для единообразной обработки ошибок в синхронном и асинхронном коде. Дальше в статье рассказывается про Temporal API, TypeScript, рантаймы и тулинг - можно быстро актуализировать знания.

## 🦮 A11y

- [Создаем WCAG-доступный DatePicker на React: как Claude пишет основу, а мы доводим до ума](https://habr.com/ru/articles/1023432/). Автор попросил Claude сгенерировать DatePicker по паттерну WAI-ARIA APG и получила рабочий каркас с ARIA-структурой и клавиатурной навигацией. Дальше начались открытия: <td> без вложенного <button> формально соответствует спецификации, но на реальных скринридерах работает хуже нативного <button>. При открытии диалога фокус не вставал на дату, aria-live="polite" накапливал очередь объявлений при быстром листании месяцев, кнопка-триггер моргала. Все починили руками. Главный вывод: AI ускорил старт на несколько часов, но доступность - территория, где каждую деталь нужно всё ещё проверять скринридером лично.
- [AI-генерируемый UI недоступен по умолчанию](https://frontendmasters.com/blog/ai-generated-ui-is-inaccessible-by-default/). Автор проверил сайдбар-навигацию, сгенерированную популярными AI-инструментами, через accessibility tree в DevTools и насчитал десять ошибок в 29 строках: нет landmark, нет заголовка, `<div onClick>` вместо `<button>`, отсутствующий `aria-expanded`, не фокусируемые элементы, иконка без имени. Причина системная: LLM оптимизирует под render tree и почти игнорирует accessibility tree - то, что читает скринридер. Статья предлагает пятислойную систему защиты: системные промпты с явными ARIA-требованиями, статический анализ (eslint-plugin-jsx-a11y), runtime-тестирование, CI-интеграция и компонентные абстракции с запечёнными контрактами доступности. Хорошо сочетается с предыдущей историей про DatePicker.
